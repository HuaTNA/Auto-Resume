# 桦（HUA）UI 需求总表
> 本文汇总截至 2026-07-17 已确认的全部 UI 要求，作为后续新增页面、重构和验收的统一依据。具体设计变量与技术约束参见 [`hua-design-system.md`](./hua-design-system.md)。

## 1. 产品体验目标

- 整体界面不能过度扁平，打开平台时应有明确的视觉吸引力、空间感和进入工作的欲望。
- 气质保持克制、自然、安静、有文学感；参考白桦、纸张、墨色以及东方书卷气质。
- 层次主要通过留白、字体、明暗、材质、边界和阴影建立，不依赖鲜艳色彩、渐变或夸张特效。
- 所有页面必须属于同一套视觉系统，不能只有首页精致、进入功能页后退回普通后台模板。
- 视觉提升不能牺牲可读性、可操作性和信息效率。

## 2. 视觉语言

### 2.1 色彩

- 页面主背景使用暖米色纸张感，而不是纯白或冷灰。
- 主要文字、图标和主操作使用墨黑色。
- 卡片、输入区域和次级模块使用有轻微差异的暖色表面，形成层次。
- 金棕色只用于细线、眉题、弱状态和装饰提示。
- 不新增系统外的彩色强调色；状态优先通过文字、位置、墨色浓度和表面差异表达。
- 禁止大面积渐变、荧光色、纯黑纯白对撞和高饱和装饰。

### 2.2 字体与排版

- 中文使用宋体气质的衬线字体，英文展示文字使用优雅的英文衬线字体。
- 页面标题应具有编辑式、书卷式气质，并与正文形成清晰字号差。
- 英文眉题使用大写和较宽字距；中文眉题使用适度字距。
- 正文保持舒适行高，避免信息挤压。
- 不使用粗重无衬线字体制造层次，不依赖大量加粗。
- 页面内容应有稳定的最大宽度和左右留白，避免内容铺满屏幕。

### 2.3 形状与材质

- 大卡片和面板使用约 `16px` 圆角。
- 按钮、标签、输入框和图标底座使用约 `6px` 圆角。
- 边框应细、淡、温和，不使用生硬深色描边。
- 阴影应柔和、偏暖、接近纸张叠放效果，不能像浮夸的玻璃拟态或电商卡片。

## 3. 全局层次系统

所有界面统一使用以下四级空间层次：

1. **画布层**：暖米色页面背景，承载整个工作区。
2. **表面层**：内容分区、表单、信息面板使用略浅表面和细边界。
3. **抬升层**：重点卡片、主功能模块和深色 Hero 使用更明确的柔和阴影。
4. **焦点层**：悬浮、选中、输入聚焦和下拉展开时增加清晰但克制的反馈。

统一要求：

- Header、Sidebar、页面内容区之间必须能一眼分辨，但不能被粗重边框割裂。
- 页面 Header 底部使用细分隔和轻微阴影，形成固定的顶部层级。
- 章节标题使用细金色引导线或同等级视觉标记，帮助扫读。
- 普通卡片使用静态浅阴影；可点击卡片悬浮时轻微上移并加强阴影。
- 深色重点模块应成为页面视觉锚点，并与浅色内容区拉开明暗层次。
- 输入框默认具有轻微内凹感；聚焦时显示清楚的边界和焦点阴影。
- 避免在同一页面给所有元素同样强度的阴影，否则会重新失去主次。

## 4. 品牌与图标

- 最终采用用户提供的 `Design Icon Set with Birch (1).zip` 中的白桦图标视觉。
- 产品品牌显示为 `HUA / 桦`，搭配 `Personal AI Workspace / 个人 AI 工作区`。
- 品牌图标使用实心、有机的白桦造型，不使用通用线性图标替代品牌图标。
- 图标资产统一存放在 `frontend/public/icons/birch/`。
- 白桦图标使用墨色填充，可带浅色树皮纹理；禁止图标渐变、描边和图标阴影。
- 白桦图标用于品牌、导航语义和装饰；搜索、箭头、关闭等纯功能符号可以使用简洁功能图形。

## 5. 布局要求

### 5.1 全局工作区

- 桌面端使用左侧 Sidebar + 右侧主内容区结构。
- Sidebar 需要清楚区分品牌、搜索入口、功能分组、当前导航和用户区域。
- 当前页面使用背景、左侧墨线或同等级方式突出；子导航层级必须清楚。
- Sidebar 可悬浮项目应有轻微位移或表面反馈，不能只有文字变色。
- 主内容区保持统一页边距、内容宽度和模块间距。
- 大模块之间至少保留 `24px` 间距。
- 移动端不得因固定宽度、阴影或装饰线造成横向滚动。

### 5.2 页面头部

- 每页包含眉题、主标题和一句简短说明。
- 页面头部与下面内容之间要有清楚但柔和的层级分隔。
- 页面级主要操作放在标题区域右侧或紧随标题内容之后，位置保持一致。

### 5.3 卡片与分区

- 卡片必须通过表面色、边界、阴影和留白区别于画布。
- 信息密度不同的卡片可使用不同抬升强度，但圆角和材质语言必须统一。
- 可点击卡片要有明确的 Hover、Focus 和按下反馈。
- 空状态不能只是大片空白，应提供说明和下一步操作。

## 6. 首页要求

- 首页定位为 `Command Center / 指挥中心`，而不是普通功能列表。
- 首屏需要有一个强视觉锚点，使用深色 Hero 承载产品价值和主要行动。
- Hero 表达“思考、构建、推进工作”的个人 AI 工作区定位。
- 首要操作包括 `Ask Copilot / 询问 AI 助手`、`Create New / 新建` 和 `Open Today / 查看今天`。
- `Create New` 下拉菜单展开后必须完整显示，不能被 Hero 或父容器裁切。
- Hero 下方展示 Today、AI Suggestions 等真正需要注意的内容，形成“今天要做什么”的入口。
- 首页既要有仪表盘效率，也要保留品牌氛围，避免像空白宣传页。

## 7. 职业模块要求

- 职业首页需要强调长期职业路径，而不是只罗列工具。
- 使用有层次的介绍卡片说明职业工作区的价值。
- Jobs & Match、Resume Studio、Applications、Career Profile、Interview Prep 等功能以统一卡片系统呈现。
- 顶部提供明确的 `Find opportunities / 寻找机会` 主操作。
- 职业子导航必须展开并清楚显示当前页面。

## 8. 职位搜索页要求

- 搜索区域使用独立抬升面板，与页面背景明确分离。
- 搜索表单包含职位或关键词、地区和主搜索按钮。
- 输入、下拉选择和按钮的高度、圆角和对齐保持一致。
- 用户输入关键词后，应提供清楚的加载、结果、空结果和失败反馈。
- 后端搜索暂时不可用时，不能只显示生硬的 `Failed to fetch`；需要提供可理解的提示，并尽可能给出可继续使用的降级结果。
- 错误提示必须属于页面视觉系统，不能使用浏览器默认样式或突兀红色块。

## 9. 设置与语言要求

- 中英文切换入口统一放在 `Settings / 设置` 页面，不放在全局 Header 或 Sidebar 中重复出现。
- 设置页应把语言作为独立分区，使用清楚的中文/英文切换控件。
- 切换结果自动保存并应用到整个工作区。
- 设置页同时维持账户、工作区管理、集成与权限等内容的分区层次。
- 语言切换控件的选中状态必须明确，并支持键盘操作和焦点状态。

## 10. 登录与认证页面要求

- 登录、注册等认证页面也必须使用同一套纸张、墨色和白桦品牌语言。
- 认证卡片与背景要有明确层次，不能是孤立的默认白色表单。
- 品牌氛围与表单效率保持平衡；输入框、按钮、错误提示遵循全局组件规则。
- 未登录用户仍需正常受到登录保护；视觉调试不能绕过或遗留认证逻辑。

## 11. 动效与交互

- 可交互表面悬浮时上移约 `4–6px`，动效约 `280–340ms`。
- 动效应自然、有一点弹性，但不能旋转、缩放、模糊或频繁晃动。
- 下拉菜单、弹层和提示必须处于正确层级，不能被父容器裁切或被相邻卡片覆盖。
- 所有主要交互必须具有 Hover、Focus、Active 和 Disabled 状态。
- 尊重 `prefers-reduced-motion`，减少动效设置开启时不执行位移动画。

## 12. 响应式与可访问性

- 桌面端、平板和移动端都应保留清楚的信息层级。
- 正文和次级文字需要保持足够对比度，不能为了“高级感”而过淡。
- 键盘用户可以看见焦点位置并操作导航、表单、语言切换和下拉菜单。
- 图标按钮必须有可理解的文本标签或无障碍名称。
- 内容缩放后不得出现主要操作被遮挡、菜单裁切或文本重叠。

## 13. 禁止项

- 不使用与白桦系统无关的通用 SaaS 蓝紫渐变。
- 不使用玻璃拟态、霓虹光、强发光、3D 塑料质感或大面积纯白。
- 不混用多套圆角、阴影、图标和字体语言。
- 不让所有卡片具有同样强度，避免页面重新变成没有主次的“卡片墙”。
- 不把语言切换重新放回全局导航。
- 不允许下拉菜单、浮层、提示条出现被裁切或覆盖的显示 Bug。
- 不允许只美化首页而遗漏功能页、设置页和认证页。

## 14. 验收清单

每次新增或修改界面，应至少确认：

- [ ] 页面遵循画布、表面、抬升、焦点四级层次。
- [ ] Header、Sidebar、章节和卡片主次清楚。
- [ ] 颜色、字体、圆角、阴影均来自统一设计系统。
- [ ] 白桦图标使用正确，没有混入无关图标风格。
- [ ] 当前导航、可点击区域和输入焦点清晰可见。
- [ ] 下拉菜单和浮层没有被裁切或遮挡。
- [ ] 中英文界面布局均正常，语言入口只在设置页。
- [ ] 搜索具备加载、结果、空状态和可理解的失败状态。
- [ ] 登录保护没有因 UI 调试受到影响。
- [ ] 桌面端和移动端没有横向溢出或文字重叠。
- [ ] 页面让用户清楚知道当前状态以及下一步可以做什么。
- [ ] 运行 `npm run lint:ui` 并通过 UI 规范检查。
